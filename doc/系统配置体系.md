# 系统配置体系

本文档介绍项目的三层配置体系：环境变量、应用配置、运行时设置，以及它们如何协作。

---

## 全局概览

项目有三个配置层次，各司其职：

```
┌─────────────────────────────────────────────────────────┐
│                    环境变量 (.env)                        │
│  部署相关的变量：API 地址、端口、版本号、权限模式等           │
│  特点：构建时注入，运行时只读，按环境（dev/prod）区分         │
└────────────────────────┬────────────────────────────────┘
                         │ 构建时注入到 import.meta.env
┌────────────────────────▼────────────────────────────────┐
│                 应用配置 (config/)                        │
│  静态配置：系统名称、颜色列表、菜单布局、功能开关等           │
│  特点：代码中写死，改了需要重新构建，不随用户操作变化          │
└────────────────────────┬────────────────────────────────┘
                         │ 提供默认值和可选项
┌────────────────────────▼────────────────────────────────┐
│              运行时设置 (store + localStorage)             │
│  用户偏好：主题色、圆角、菜单展开/折叠、语言等               │
│  特点：运行时可变，持久化到 localStorage，跨会话保留         │
└─────────────────────────────────────────────────────────┘
```

## 一、环境变量

### 1.1 文件结构

```
.env                  # 通用变量（所有环境共享）
.env.development      # 开发环境专用
.env.production       # 生产环境专用
env.d.ts              # TypeScript 类型声明
```

Vite 的加载规则：

- `.env` 始终加载
- `.env.development` 仅 `vite dev` 时加载
- `.env.production` 仅 `vite build` 时加载
- 后加载的文件会覆盖先加载的同名变量

### 1.2 变量清单

**`.env`（通用）：**

| 变量                    | 作用            | 示例值                 |
| ----------------------- | --------------- | ---------------------- |
| `VITE_VERSION`          | 应用版本号      | `1.0.0`                |
| `VITE_PORT`             | 开发服务器端口  | `3006`                 |
| `VITE_BASE_URL`         | 部署基础路径    | `/`                    |
| `VITE_ACCESS_MODE`      | 权限模式        | `frontend` / `backend` |
| `VITE_WITH_CREDENTIALS` | 跨域携带 Cookie | `false`                |
| `VITE_LOCK_ENCRYPT_KEY` | 锁屏加密密钥    | `s3cur3k3y4zhao`       |

**`.env.development`（开发）：**

| 变量                 | 作用                           |
| -------------------- | ------------------------------ |
| `VITE_API_URL`       | API 基础路径（`/` 表示用代理） |
| `VITE_API_PROXY_URL` | 代理目标地址（解决跨域）       |
| `VITE_DROP_CONSOLE`  | 是否删除 console               |

**`.env.production`（生产）：**

| 变量                | 作用                                |
| ------------------- | ----------------------------------- |
| `VITE_API_URL`      | API 完整地址（直接请求后端）        |
| `VITE_DROP_CONSOLE` | 是否删除 console（生产通常 `true`） |

### 1.3 命名规则

所有客户端可用的变量必须以 `VITE_` 开头。这是 Vite 的安全机制——防止服务端密钥泄露到前端。

```env
# ✅ 前端可访问
VITE_API_URL = /api

# ❌ 前端无法访问（没有 VITE_ 前缀）
SECRET_KEY = abc123
```

### 1.4 在代码中使用

通过 `import.meta.env` 访问：

```ts
// 读取 API 地址
const apiUrl = import.meta.env.VITE_API_URL

// 读取权限模式
const accessMode = import.meta.env.VITE_ACCESS_MODE // 'frontend' 或 'backend'

// Vite 内置变量
const isDev = import.meta.env.DEV // true = 开发环境
const isProd = import.meta.env.PROD // true = 生产环境
```

### 1.5 TypeScript 类型

`env.d.ts` 中声明自定义全局变量：

```ts
declare const __APP_VERSION__: string // 由 Vite 构建时从 VITE_VERSION 注入
```

`import.meta.env` 的 VITE\_ 变量由 Vite 自动生成类型，无需手动声明。

### 1.6 实际应用示例

**API 请求（`src/utils/http/`）：**

```ts
// 根据环境变量决定 API 地址
const baseURL = import.meta.env.VITE_API_URL
```

**权限系统（`src/hooks/core/useAppMode.ts`）：**

```ts
// 根据环境变量决定权限模式
const accessMode = import.meta.env.VITE_ACCESS_MODE

// frontend 模式：前端路由配置中定义权限
// backend 模式：后端 API 返回菜单和权限
```

**开发代理（Vite 自动读取）：**

```
开发时请求 /api/users
  → Vite 代理转发到 VITE_API_PROXY_URL/api/users
  → 解决跨域问题
```

## 二、应用配置（静态）

### 2.1 文件结构

```
src/config/
├── index.ts              # 主配置（系统信息、颜色、布局、主题列表）
├── setting.ts            # 设置默认值（SETTING_DEFAULT_CONFIG）
├── assets/
│   └── images.ts         # 配置用图片资源
└── modules/
    ├── fastEnter.ts      # 快速入口配置
    └── headerBar.ts      # 顶部栏功能开关配置
```

### 2.2 主配置 (`config/index.ts`)

这是应用的核心配置对象，类型为 `SystemConfig`，通过 `Object.freeze()` 冻结防止运行时修改：

```ts
const appConfig: SystemConfig = {
  systemInfo: { name: '朝而往' },              // 系统名称
  systemThemeStyles: { ... },                   // 主题样式映射
  settingThemeList: [ ... ],                    // 可选主题列表（Light/Dark/System）
  menuLayoutList: [ ... ],                      // 菜单布局列表（Left/Top/Mixed/Dual）
  themeList: [ ... ],                           // 菜单风格列表（Design/Dark/Light）
  darkMenuStyles: [ ... ],                      // 暗黑模式下的菜单样式
  systemMainColor: ['#5D87FF', '#B48DF3', ...], // 预设主题色
  fastEnter: fastEnterConfig,                   // 快速入口
  headerBar: headerBarConfig,                   // 顶部栏功能开关
}

export default Object.freeze(appConfig)
```

**为什么用 `freeze`**：这些是构建时确定的静态配置，不应被运行时意外修改。如果需要修改，必须改源码重新构建。

**使用方式**：

```ts
import AppConfig from '@/config'

// 读取系统名称
AppConfig.systemInfo.name // '朝而往'

// 读取预设颜色列表
AppConfig.systemMainColor // ['#5D87FF', '#B48DF3', ...]

// 读取菜单布局选项
AppConfig.menuLayoutList // [{ name: 'Left', value: MenuTypeEnum.LEFT, ... }, ...]
```

### 2.3 顶部栏功能开关 (`config/modules/headerBar.ts`)

控制顶部栏各功能按钮是否可用（代码级别开关）：

```ts
export const headerBarConfig: HeaderBarFeatureConfig = {
  menuButton:    { enabled: true, description: '...' },  // 菜单折叠按钮
  refreshButton: { enabled: true, description: '...' },  // 刷新按钮
  breadcrumb:    { enabled: true, description: '...' },  // 面包屑
  globalSearch:  { enabled: true, description: '...' },  // 全局搜索
  fullscreen:    { enabled: true, description: '...' },  // 全屏
  notification:  { enabled: true, description: '...' },  // 通知
  language:      { enabled: true, description: '...' },  // 语言切换
  settings:      { enabled: true, description: '...' },  // 设置面板
  themeToggle:   { enabled: true, description: '...' },  // 主题切换
  ...
}
```

设为 `false` 则该功能完全不显示，用户无法在设置面板中开启。

### 2.4 功能显示的双重控制

顶部栏功能受两层控制：

```
headerBar 配置（代码级开关）  AND  settingStore（用户级开关）
         ↓                              ↓
    enabled: true/false          showXxx: true/false
         ↓                              ↓
         └────────── AND ──────────────┘
                       ↓
               最终是否显示
```

`useHeaderBar` 组合函数中实现：

```ts
// 两层都为 true 才显示
const shouldShowMenuButton = computed(() => {
  return isFeatureEnabled('menuButton') && showMenuButton.value
  //       ↑ headerBar 配置           ↑ store 用户设置
})
```

这意味着：

- 如果 `headerBar.enabled = false`，用户无法通过设置面板开启
- 如果 `headerBar.enabled = true`，用户可以在设置面板中自由开关

### 2.5 设置默认值 (`config/setting.ts`)

`SETTING_DEFAULT_CONFIG` 定义了所有用户可调设置的默认值：

```ts
export const SETTING_DEFAULT_CONFIG = {
  menuType: MenuTypeEnum.LEFT, // 默认菜单布局
  menuOpenWidth: 200, // 默认菜单宽度
  menuOpen: true, // 默认展开
  systemThemeType: SystemThemeEnum.AUTO, // 默认自动主题
  menuThemeType: MenuThemeEnum.DESIGN, // 默认设计风格菜单
  systemThemeColor: '#5D87FF', // 默认主题色
  showCrumbs: true, // 默认显示面包屑
  showWorkTab: true, // 默认显示标签栏
  showNprogress: true, // 默认显示进度条
  customRadius: '0.75', // 默认圆角
  pageTransition: 'slide-left', // 默认页面动画
  // ... 更多
}
```

**作用**：给 store 中的 `ref()` 提供初始值。用户修改设置后，store 中的值会变化并持久化，但这里的默认值不变。

## 三、运行时设置（Store）

### 3.1 整体架构

```
Pinia Store (settingStore)
├── 状态 (ref)
│   ├── menuType          菜单布局类型
│   ├── menuOpenWidth     菜单宽度
│   ├── systemThemeColor  主题色
│   ├── customRadius      圆角
│   ├── showCrumbs        面包屑开关
│   ├── pageTransition    页面动画
│   └── ... 更多
├── 计算属性 (computed)
│   ├── getMenuTheme      当前菜单主题配置
│   ├── isDark            是否暗色模式
│   └── getMenuOpenWidth  菜单宽度（带单位）
├── 方法 (function)
│   ├── setThemeColor     设置主题色 → 调用 setArcoThemeColor
│   ├── setCustomRadius   设置圆角 → 设置 CSS 变量
│   ├── setCrumbs         切换面包屑
│   └── ... 更多
└── 持久化配置
    ├── key: 'setting'    localStorage 键名
    └── storage: localStorage
```

### 3.2 状态定义

每个设置项都是一个 `ref()`：

```ts
const systemThemeColor = ref(SETTING_DEFAULT_CONFIG.systemThemeColor) // '#5D87FF'
const customRadius = ref(SETTING_DEFAULT_CONFIG.customRadius) // '0.75'
const showCrumbs = ref(SETTING_DEFAULT_CONFIG.showCrumbs) // true
```

初始值来自 `SETTING_DEFAULT_CONFIG`。如果 localStorage 中有持久化的值，Pinia 持久化插件会自动覆盖。

### 3.3 持久化机制

Store 配置了 `pinia-plugin-persistedstate`：

```ts
export const useSettingStore = defineStore(
  'settingStore',
  () => {
    // ... 状态和方法
  },
  {
    persist: {
      key: 'setting',
      storage: localStorage,
    },
  },
)
```

**持久化流程**：

```
用户修改设置
  → store 中的 ref 值变化
    → 持久化插件自动序列化整个 store 到 localStorage
      → 键名由 StorageKeyManager 生成：sys-v{version}-setting
        → 例：sys-v1.0.0-setting

页面加载
  → Pinia 创建 store，ref 使用默认值
    → 持久化插件从 localStorage 读取 sys-v1.0.0-setting
      → JSON.parse 后覆盖 ref 的默认值
        → 用户上次的设置自动恢复
```

### 3.4 版本化存储键

`StorageKeyManager` 负责生成带版本号的 localStorage 键名：

```ts
// 键名格式：sys-v{version}-{storeId}
// 例如：sys-v1.0.0-setting、sys-v1.0.0-user
```

**为什么需要版本号**：应用升级后，store 结构可能变化（新增/删除/重命名字段）。版本化键名可以：

1. 隔离不同版本的数据，避免结构不兼容
2. 自动迁移：检测到旧版本数据时，复制到新版本键名下

```ts
// StorageKeyManager.getStorageKey 的逻辑：
// 1. 检查 sys-v1.0.0-setting 是否存在 → 存在则直接用
// 2. 不存在 → 查找旧版本键（如 sys-v0.9.0-setting）
// 3. 找到旧数据 → 复制到 sys-v1.0.0-setting → 返回新键
```

### 3.5 设置面板（UI 层）

`zhao-settings-panel/index.vue` 是设置面板的 UI，它读取 store 状态并调用 store 方法：

```html
<!-- 面包屑开关 -->
<a-switch
  :model-value="settingStore.showCrumbs"       <!-- 读取 store 状态 -->
  @change="settingStore.setCrumbs"              <!-- 调用 store 方法 -->
/>

<!-- 主题色选择 -->
<div v-for="color in mainColors"
  @click="settingStore.setThemeColor(color)"    <!-- 点击 → 更新 store + 应用到 DOM -->
/>

<!-- 圆角滑块 -->
<a-slider
  :model-value="Number(settingStore.customRadius)"
  @change="(v) => settingStore.setCustomRadius(String(v))"
/>
```

### 3.6 设置生效的方式

不同设置生效的方式不同：

| 设置               | 生效方式      | 说明                                                                 |
| ------------------ | ------------- | -------------------------------------------------------------------- |
| `showCrumbs`       | Vue 响应式    | `v-if="settingStore.showCrumbs"` 直接控制 DOM 显示                   |
| `menuType`         | Vue 响应式    | `computed` 属性变化触发布局组件重新渲染                              |
| `systemThemeColor` | CSS 变量 + JS | 调用 `setArcoThemeColor()` 修改 `body.style` 上的 CSS 变量           |
| `customRadius`     | CSS 变量 + JS | 调用 `setProperty` 修改 `--custom-radius`，main.css 映射到 Arco 变量 |
| `pageTransition`   | Vue 响应式    | `<Transition :name="transitionName">` 绑定 computed 属性             |
| `menuOpenWidth`    | Vue 响应式    | `:style="{ width: sidebarWidth }"` 绑定 computed 属性                |

## 四、三层配置的关系

```
环境变量 (.env)
  │
  │  构建时注入到 import.meta.env
  │  运行时只读，不可修改
  │
  ▼
应用配置 (config/)
  │
  │  Object.freeze() 冻结
  │  提供：颜色选项、布局选项、默认值、功能开关
  │
  ├──────────────────────────┐
  │                          │
  ▼                          ▼
Store 默认值              设置面板选项列表
  │                          │
  │  ref(SETTING_DEFAULT_CONFIG.xxx)
  │                          │
  ▼                          ▼
Pinia Store (settingStore)  设置面板 UI
  │                          │
  │  运行时状态               │  用户交互
  │  持久化到 localStorage    │
  │                          │
  ├──────────────────────────┘
  │
  ▼
页面渲染
  │
  ├── Vue 响应式（v-if、:style、computed）
  ├── CSS 变量注入（body.style.setProperty）
  └── Arco Design 组件自动响应
```

### 4.1 一个设置的完整生命周期

以「主题色」为例：

```
1. 定义阶段（代码）
   config/index.ts → systemMainColor: ['#5D87FF', '#B48DF3', ...]
   config/setting.ts → systemThemeColor: '#5D87FF'

2. 初始化阶段（页面加载）
   main.ts → initStore(app)
     → Pinia 创建 settingStore，ref 使用默认值 '#5D87FF'
     → 持久化插件从 localStorage 读取，如果用户之前改过则覆盖
   main.ts → initializeTheme()
     → 读取 settingStore.systemThemeColor
     → 调用 setArcoThemeColor() 设置 CSS 变量到 body

3. 用户交互阶段
   用户点击紫色色块 → settingStore.setThemeColor('#B48DF3')
     → store 中 systemThemeColor 更新为 '#B48DF3'
     → 持久化插件自动写入 localStorage（sys-v1.0.0-setting）
     → setArcoThemeColor('#B48DF3') 更新 body 上的 CSS 变量
     → 页面上所有使用这些变量的元素立即变色

4. 下次加载
   从 localStorage 恢复 '#B48DF3' → 主题色自动生效
```
